I have completed V1 - V5 of the PageCheck subtasks. The final version of the program that I am submitting is V5, the get creative subtask.

After completing subtasks V1 - V4 I wasn't entirely satisified with the functionality of the program. The reason was that many web servers do not return the header corresponding to the last time the page was modified. This made the program return the Unix epoch, Jan. 1, 1970, as the last modified date for these pages. This made the program ineffectual for many web pages depending on the configuration of the hosting web server. Interestingly enough the web server which hosts the www2.hawaii.edu domain does not return this header and therefore I couldn't use my original PageCheck program on my own web pages.

I decided to resurrect an idea I had used in a Perl program that checks for changes in web pages by storing the algorithmic hash of the contents of the web page and then compares that hash to the latest hash when the program is run again. This avoids the problem with the last modified header not being supplied and can therefore be applied to a wider range of web pages.

It was interesting to write this program in Java, as my Java version relies on the extensive class libraries while my Perl version pipes data through other programs (such as md5sum for the hash and mail for mailing).

So in summary my V5 version:

  • accepts command line input of one or more files
  • each file contains PageCheck job information that can specify multiple urls to check with multiple recipients
  • changes are indicated by a change in the algorithmic hash
I have tried to think of all the exceptional conditions that can exist and program accordingly. These include:

problemreaction
input file or files not provided program outputs proper usage
input file cannot be read program outputs message
input file not in correct format / missing data program displays what is missing from the input file
data file not in correct format program outputs message
url cannot be fetched programs sends failed: url in message
hash data cannot be saved message includes failure to save new hash values

Of course the only true test of this would be to have a substantial number of users bang on the program for a while.
